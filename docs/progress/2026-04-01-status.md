# 2026-04-01 项目推进状态与提交建议

## 本轮 review 结论

当前仓库已经具备“可验证的最小业务闭环原型”，但尚未具备“可对外宣称完整实现 spec”的完成度。

更准确的判断是：

- 进入了 `Phase 1 / Phase 2` 交界处
- 已有一条默认链式代理 happy path 可以用固定夹具回放
- 真实运行时集成、完整校验与 API 对外面仍未落地

## 已完成内容

### 1. 基础工程

- Go module 与目录骨架已建立
- `internal/api`、`internal/config`、`internal/store`、`internal/subconverter`、`internal/service` 包路径已固定
- 旧实现已归档，可继续作为参考但不再是权威

### 2. 测试基线

- `3pass-ss2022-test-subscription` 已不再只是 `subconverter` 夹具，而是最小完整流程样例
- 已固定 `stage1-convert` / `generate` 请求与成功响应 golden
- 已固定规范长链接编码前载荷与最终订阅 YAML golden

### 3. 服务层原型

- 已能从固定 `3-pass` 结果推导默认 `stage2Init`
- 已能对固定 `stage2Snapshot` 生成规范长链接
- 已能基于固定 `full-base` 结果渲染默认链式代理订阅

## 当前阻塞项

以下问题使得当前版本不宜按“阶段完成”口径提交：

### P1. `subconverter` 仍未真实集成

- `internal/subconverter` 只有包说明，没有真实 HTTP 调用实现
- 目前所有流程建立在静态测试夹具之上，还不能代表真实服务行为

### P1. 区域识别尚未按 spec 实现

- spec 要求区域识别基于当前实际生效配置文件中的 6 条完整规则
- 当前原型使用代码内置正则匹配落地节点名，只能证明 happy path，不足以证明规则已收口

### P1. 端口转发输入校验远未完成

- spec 要求严格的 `server:port` 逐行校验、规范化与重复判定
- 当前实现只保留了最小拆分与原值去重逻辑，未满足契约

### P2. 错误模型仍是原型态

- `messages[]`、`blockingErrors[]` 的完整错误码、`scope`、`context` 语义还未实现
- `restrictedModes`、恢复冲突、rowset mismatch 等失败路径尚未形成稳定输出

### P2. API / Store / Config / Server 均未接通

- `POST /api/stage1/convert`
- `POST /api/generate`
- 短链接索引
- 可配置限制项
- 启动装配

以上都仍未落地为真实运行路径

## 本轮文档整理结论

为避免后续提交说明继续失真，本轮建议采用以下边界：

- 可以提交：测试资产补齐、服务层 happy path 原型、推进状态文档
- 不应宣称：完整 `subconverter` 集成、完整业务规则实现、完整 API 落地

## 建议提交主题

若本轮准备提交，建议把主题收敛为：

- 固化 `3-pass` 到最小完整流程测试资产
- 增加服务层默认链式 happy path 原型与 golden test
- 补齐当前推进状态与提交边界文档

## 提交前自检

本轮建议至少确认以下几点：

1. 提交说明明确写“prototype”或“happy path”
2. 不把 `internal/api`、`internal/subconverter`、`internal/store` 描述为已完成
3. 测试结果写明当前仅验证 `go test ./...`
4. 后续任务按“先集成，再收口规则，再接 API”排序

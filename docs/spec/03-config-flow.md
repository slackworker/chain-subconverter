# 03 - 配置生成主流程（解析 → 合成 → 修改 → 输出）

> 依赖：本章严格按 [02-prerequisites](02-prerequisites.md) 的输入模型与约束执行，尤其是：
>
> - **完整配置判定标准**（02 的 1.2）
> - **两份输入源同时存在时的基准配置选择规则**（02 的 1.5）
> - **修改方式与协议约束**（02 的「四、修改方式」与 3.4 约束）

---

## 总览：四阶段流水线

本项目对外体现为一条固定的四阶段流水线：

1. **阶段 1：解析**（拿信息）  
   解析用户提供的“中转信息输入”和“落地信息输入”，抽取：
   - 是否包含**完整 Mihomo 配置**
   - 可用的**节点集合**（中转节点/落地节点）
   - 端口转发所需的**中转机 `server:port` 列表**
2. **阶段 2：合成**（得到基准完整配置）  
   选择或生成一份**基准（完整）Mihomo 配置**，并把缺失的中转/落地节点拼接进去，得到可修改的“完整配置骨架 + 节点集合”。
3. **阶段 3：修改**（按用户指定方式改写）  
   按用户指定的修改方式，对落地节点进行链式代理与/或端口转发改写，同时生成必要的 `proxy-groups` / `proxy-providers` 引用。
4. **阶段 4：输出**  
   输出最终完整 YAML（以及可选的订阅短链接），见 [04-api-output](04-api-output.md)。

---

## 阶段 1：解析（输入 → 结构化信息）

### 1.1 输入分为两份（Web 前端的主交互）

- **中转信息输入**：可能是订阅/YAML/节点-only/节点链接/端口转发中转机 `server:port`
- **落地信息输入**：可能是订阅/YAML/节点-only/节点链接/填表（SOCKS5）

解析产物（统一结构，不要求实现细节一致，但语义必须一致）：

- `transit.full_config?`：是否解析得到完整配置（见 02 的 1.2）
- `landing.full_config?`：是否解析得到完整配置（见 02 的 1.2）
- `transit.nodes[]`：中转候选节点（Mihomo proxy 格式）
- `landing.nodes[]`：落地/出口节点集合（Mihomo proxy 格式）
- `forward.relays[]`：端口转发中转机列表（`{ server, port }`）

### 1.2 落地节点的“选择/确认”

落地节点必须由用户指定/确认（自动识别只能作为默认建议）：

- **自动识别（建议）**：按节点名关键字（`落地` / `Landing` / `出口` / `exit` 等）
- **手动指定（必备）**：从解析出的节点列表中勾选

> 说明：端口转发与链式代理都需要明确“哪些节点是落地节点”，否则会导致误改写普通机场节点。

---

## 阶段 2：合成（确定基准配置，并拼接缺失节点）

### 2.1 基准配置选择（两份输入源的决策）

基准配置选择规则以 02 的 1.5 为准：

- 两份都没有完整配置 → **合并全部节点** → 走**内置订阅转换**生成完整配置骨架
- 只有一份有完整配置 → 以该份为基准
- 两份都有完整配置 → **必须由用户选择**以哪份为基准（否则报错）

### 2.2 拼接规则（把“中转/落地节点”写入基准配置）

合成阶段必须保证后续“修改”可引用到以下信息：

- `proxies:` 中包含：
  - 用户确认的落地节点（若基准中没有则追加；有则复用）
  - 用户提供的中转节点（若修改方式需要且基准中没有则追加；有则复用）
- `proxy-groups:` 与 `rules:` 必须存在（若基准为内置订阅转换产物则天然满足；若用户提供的 YAML 不满足“完整配置判定标准”，必须回到“内置订阅转换”）

同名冲突处理（必要时）：

- 若追加节点时发生同名冲突，必须先重命名再写入，避免覆盖与歧义（命名规则见 02 的 3.4）。

---

## 阶段 3：修改（链式 / provider / 端口转发）

### 3.1 修改前校验（必须在前端提示 & 后端强校验）

- **链式代理（dialer-proxy）**：落地节点仅允许 **ss / SOCKS5**（不支持 reality）
- **端口转发**：所有协议只替换 `server`/`port`；reality 必须走“复制新节点再改写”
- **链式 + 端口转发同时启用**：仅对 **ss / SOCKS5** 落地节点生效

### 3.2 修改方式 A：链式代理（新增 dialer 组 + 给落地节点加 dialer-proxy）

目标：为每个落地节点生成一个 dialer 策略组，并在落地节点上添加 `dialer-proxy`。

#### A1. 选项一：基于 `proxies` 节点创建 dialer 组（`proxy-groups.proxies`）

适用：中转候选节点以 `proxies` 形式存在（来自基准配置或被拼接进入）。

1. 在 `proxy-groups` 中新增 `"<落地节点名称> dialer"`：
   - `type`: 默认 `url-test`（也可 `select`）
   - `url/interval/tolerance`: 仅 `url-test` 需要（与旧文一致）
   - `proxies`: 由用户在 UI 中选择的中转节点名列表
2. 防递归：dialer 组成员 **不得包含该落地节点自身**
3. 改写落地节点：
   - `dialer-proxy: "<落地节点名称> dialer"`
   - 节点名改为 `原名称 + 空格 + "链式"/"Chain"`（按是否含中文选择后缀）

> UI 选择策略（区域分类/手动选择）属于前端实现细节，但必须满足“可选节点集合来自合成后的基准配置 + 用户拼接的中转节点”，并确保落地节点自身不可被选中。

#### A2. 选项二：基于 `proxy-providers` 创建 dialer 组（`proxy-groups.use`）

适用：基准配置中存在 `proxy-providers`，用户希望复用某个 provider。

1. 用户从已有 `proxy-providers` 单选一个 provider（如 `provider1`）
2. 在 `proxy-groups` 中新增 `"<落地节点名称> dialer"`：
   - `type: select`
   - `use: [provider1]`
3. 改写落地节点添加 `dialer-proxy: "<落地节点名称> dialer"` 并按规则改名

约束：若基准配置无 `proxy-providers`，该选项必须隐藏/禁用。

### 3.3 修改方式 B：端口转发（替换 server/port）

目标：将落地节点的 `server`/`port` 替换为一个或多个中转机 `server:port`。

- `forward.relays[]` 可包含多个中转机；具体“一对一还是多份输出”属于产品策略：
  - **单份输出**：只允许用户选择一个中转机
  - **多份输出**：每个中转机生成一份输出配置（更贴近“订阅多条”）

节点改写策略（必须遵守 02 的 3.4）：

- **ss / SOCKS5**：直接改写原节点（按需要改名/加 dialer-proxy/替换 `server`/`port`）
- **reality**：保留原节点；复制生成新节点并在新节点上替换 `server`/`port`，新节点命名为 `原名称 + 空格 + "转发"/"Forward"`

### 3.4 同时启用（链式 + 端口转发）

对 **ss / SOCKS5** 落地节点：

- 先确保 dialer 组已生成且不包含落地自身
- 在“链式改名后的落地节点”上：
  - 保持/添加 `dialer-proxy`
  - 同时替换 `server`/`port` 为选定中转机 `server:port`

对 reality 落地节点：禁止链式；仅允许端口转发（复制新节点）。

---

## 阶段 4：输出（完整 YAML / 订阅短链接）

输出必须是“可直接供 Mihomo 使用的完整配置”，并支持：

- 直接返回 YAML（下载/复制）
- 可选：生成订阅短链接（`/subscription/<id>.yaml`），契约见 [04-api-output](04-api-output.md)

---

## 数据流（四阶段）

```mermaid
flowchart TB
  subgraph I[输入（两份信息）]
    T[中转信息输入]
    L[落地信息输入]
  end

  subgraph S1[阶段1：解析]
    P[解析两份输入<br/>抽取：完整配置? 节点集合 relay:server:port]
    SEL[落地节点选择/确认<br/>自动建议 + 手动勾选]
  end

  subgraph S2[阶段2：合成]
    BASE[选择/生成基准完整配置<br/>(必要时内置订阅转换)]
    MERGE[拼接缺失的中转/落地节点<br/>保证 proxies/proxy-groups/rules 完整]
  end

  subgraph S3[阶段3：修改]
    M1[链式代理<br/>dialer 组 + dialer-proxy]
    M2[端口转发<br/>替换 server/port]
  end

  subgraph O[阶段4：输出]
    OUT[完整 Mihomo YAML<br/>（可选订阅短链接）]
  end

  T --> P
  L --> P
  P --> SEL
  SEL --> BASE
  BASE --> MERGE
  MERGE --> M1
  MERGE --> M2
  M1 --> OUT
  M2 --> OUT
```

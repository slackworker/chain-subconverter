# 03 - 修改完整配置（链式代理 / 端口转发）

> 前置：必须先得到“完整配置”（通用配置 + 节点），见 [02-generate-complete-config](02-generate-complete-config.md)。

本章只描述：在“完整配置”基础上，如何按用户指定方式进行改写，生成最终可直接供 Mihomo 使用的 YAML。

## 输入（修改阶段必须拿到的信息）

修改阶段的输入（概念层面）至少包含：

- **`complete_config`**：一份完整配置（通用配置 + 节点），来源见 02。
- **`selected_transit_names[]`**：用户确认的中转节点名称列表。
- **`modification_mode`**：`chain` / `port_forward` / `chain_and_port_forward`。
- **落地节点确认**：用户明确指定哪些节点是落地节点（例如 `selected_landing_names[]`）。
- **`landing_included_in_transit`**：是否允许从中转输入来源中同时选出落地节点；该信息主要用于保留业务语义与前端回显，不应用于替代显式角色列表。
- **链式代理选择（当模式包含链式）**：用户为“每个落地节点”指定其 dialer 组成员（中转节点集合或 provider）。
- **端口转发信息（当模式包含端口转发）**：
  - `forward_relays[]`：可用的 `server:port` 列表（或等价结构）
  - 以及用户的“转发映射”：哪些落地节点要转发到哪个 relay（全局同一个 relay 或按节点分别指定均可）

> 说明：你提出的“用户指定方式就是谁落地、谁中转、谁转发”，本章把它落实为三类输入：
>
> - **角色划分**：中转节点清单 + 落地节点清单
> - **转发映射**：落地节点 → relay 的映射（或一个全局 relay）
> - **保留语义**：落地是否已包含在中转输入中

## 修改方式（总览）

- **链式代理（Chain）**：为落地节点增加 `dialer-proxy`，其连接经由中转节点发起。
- **端口转发（Port Forward）**：将落地节点的 `server`/`port` 替换为端口转发服务信息中的 `server:port`。
- **链式 + 端口转发**：对满足约束的落地节点同时应用两种改写。

> 具体输入字段与 API 契约见 [04-output-and-api](04-output-and-api.md)。

## 修改前校验（必须强校验）

- **链式代理**：落地节点仅允许 **ss / SOCKS5**（不支持 reality）。
- **端口转发**：所有协议仅替换 `server`/`port`；reality 必须走“复制新节点再改写”的策略。
- **链式 + 端口转发**：仅对 **ss / SOCKS5** 落地节点生效；reality 仍只允许端口转发。

## 节点命名规则（最低要求）

为避免覆盖原节点与造成歧义，修改阶段必须保证节点名唯一，并体现改写类型：

- **链式**：追加后缀 `Chain` 或 `链式`（是否含中文可由实现自动判断；规则可实现细化，但必须稳定）。
- **端口转发**：追加后缀 `Forward` 或 `转发`。
- **链式 + 端口转发**：建议追加组合后缀 `Chain Forward` 或 `链式 转发`。

> reality 节点在端口转发时必须复制出新节点，因此更需要明确后缀以区分原节点与转发节点。

## 修改流程

### 步骤 1：定位落地节点（按用户确认）

- 若 `selected_landing_names[]` 中有任一节点在配置中不存在：必须报错（防误操作）。
- 落地节点集合确定后，后续改写**只作用于这些节点**。

### 步骤 2：链式代理（当模式包含链式）

目标：为每个落地节点创建一个 dialer 组，并在落地节点上增加 `dialer-proxy` 指向该组。

对每个落地节点 `L`：

1. 创建（或更新）dialer 组：`"<L> dialer"`（如冲突必须重命名，避免覆盖）。
2. dialer 组的成员来源二选一（由用户输入决定）：
   - **基于节点名列表**：`proxy-groups.<dialer>.proxies = ["TransitA", "TransitB", ...]`
   - **基于 provider**：`proxy-groups.<dialer>.use = ["provider1"]`（此时 `type` 建议为 `select`）
3. 防递归：dialer 组成员**不得包含该落地节点自身**（否则会自我套娃）。
4. 改写落地节点：
   - 写入 `dialer-proxy: "<L> dialer"`
   - 按命名规则改名（例如追加 `Chain/链式`，或在“链式+转发”模式下追加组合后缀）

### 步骤 3：端口转发（当模式包含端口转发）

目标：把“落地节点实际连接的 `server`/`port`”替换为 relay 的 `server:port`。

对每个落地节点 `L`：

1. 确定 `L` 的目标 relay（来自用户的“转发映射”）：得到 `{ server, port }`。
2. 如果 `L` 为 reality 节点：
   - **必须复制**出新节点 `L'`
   - 在 `L'` 上替换 `server`/`port`
   - `L` 原节点保持不变
   - `L'` 按命名规则追加 `Forward/转发`（或组合后缀）
3. 如果 `L` 非 reality：
   - 允许直接在原节点上替换 `server`/`port`
   - 并按命名规则追加 `Forward/转发`（或组合后缀）

### 步骤 4：一致性检查（修改后必须满足）

- 修改后配置仍必须满足“完整配置”定义（通用配置字段存在 + 节点可被引用）。
- 若新增 dialer 组、复制节点导致 `proxy-groups` 或 `proxies` 出现重名引用：必须报错或自动消歧（不可静默覆盖）。

## 输出约束

- 输出必须仍是“完整配置”（通用配置 + 节点），并可直接供 Mihomo 使用。

# 02 - 生成完整配置（通用配置 + 节点，含模板与 subconverter）

> 本章定义“如何得到可用于后续改写的**完整配置**”。后续链式代理/端口转发只在“完整配置”上进行（见 [03-modify-config](03-modify-config.md)）。

## 术语（本章口径）

- **节点集合（Nodes）**：用户提供/解析得到的中转节点与落地节点，最终写入 `proxies`（以及可选 `proxy-providers`）。
- **通用配置（GeneralConfig）**：使 Mihomo 可运行并支撑后续改写的最小非节点结构，必须包含：
  - `mixed-port` 或 `port`
  - `mode`
  - `external-controller`
  - `proxy-groups`
  - `rules`
- **模板（Template）**：用于生成/补齐通用配置的来源；有默认模板，也允许用户自定义模板。
- **完整配置（CompleteConfig）**：`节点集合 + 通用配置`，并满足：
  - 顶层包含通用配置最小字段集合
  - 顶层存在 `proxies`（或 `proxy-providers`），且最终节点集合同时覆盖**中转节点**与**落地节点**
  - 可直接作为后续修改的输入（见 [03-modify-config](03-modify-config.md)）
- **subconverter**：用于“节点（可多份）+ 模板 + 参数 → 生成完整配置”。

---

## 输入

输入在概念上分为四类（具体 API 结构见 [04-output-and-api](04-output-and-api.md)）：

- **中转信息输入（必填其一）**：可能包含“完整配置（含通用配置）”或“仅节点”。允许多份来源。
- **落地信息输入（可选）**：通常为“仅节点”（可多份）。
- **模板（可选）**：若需要生成/补齐通用配置，可使用默认模板或用户自定义模板。
- **端口转发服务信息（可选）**：仅端口转发模式需要（例如 `server:port` 列表），本章只负责把它结构化并传递给后续修改阶段。

---

## 阶段 1：解析（输入 → 结构化信息）

本章不规定实现细节，但要求输出语义一致的结构化结果：

- `transit.general_config?`：中转输入是否包含通用配置（本章的核心检测结果）
- `transit.nodes[]`：从中转输入解析到的节点集合（Mihomo proxy 格式或等价的内部表示）
- `landing.nodes[]`：从落地输入解析到的节点集合
- `template.selected`：默认模板 / 用户自定义模板 / 未选择（仅当允许未选择时）
- `subconverter.params`：可选参数集合（用于路径 B）

> 注意：端口转发的 `server:port` 不属于“节点”，应单独解析成结构化的 relay 列表，交给后续修改阶段处理。

## 自动检测：中转信息是否包含通用配置

程序必须对“中转信息输入”的解析结果做分流判断：

- **包含通用配置**：意味着用户提供的信息可视为“完整配置候选”（至少含通用配置；节点是否齐全由合并规则补齐）。
- **仅节点**：不包含通用配置，无法直接作为修改基础，必须通过“模板 + subconverter”生成通用配置，才得到完整配置。

### 判定规则（必须）

当且仅当解析得到的 YAML 顶层对象同时满足以下条件时，判定 `transit.general_config? = true`：

- 存在 `mixed-port` **或** `port`
- 存在 `mode`
- 存在 `external-controller`
- 存在 `proxy-groups`
- 存在 `rules`

否则判定为 `false`（视为“仅节点”或“不含通用配置”的输入）。

> 说明：本判定只用于识别“是否包含通用配置”，不要求 `proxies/proxy-providers` 一定存在；节点是否齐全由后续“节点合并/生成路径”处理。

## 生成完整配置的两条路径

## 阶段 2：节点合并与命名冲突（两条路径共用）

无论走哪条路径，系统都需要把多来源节点合并为可引用的最终集合：

- **来源合并**：
  - 中转输入解析得到的节点（若有）
  - 落地输入解析得到的节点（若有）
- **命名唯一性**：写入 `proxies` 的节点 `name` 必须唯一；若冲突，必须重命名（规则可在 03 中细化，但此处要求“不可覆盖/不可歧义”）。
- **角色确认**：哪些是中转节点、哪些是落地节点，必须由用户指定/确认（自动识别只能作为默认建议）；该信息用于后续修改阶段（见 03）。

### 路径 A：使用用户自己的通用配置（当且仅当检测到“包含通用配置”）

适用：用户提供的中转信息里**包含通用配置**，且用户选择“使用自己的通用配置”。

产物：通过**在该配置上整合/追加节点信息**，得到用于后续修改的完整配置。

处理步骤（要求语义）：

1. 选择一份“通用配置来源”作为基底（当中转输入解析出多份包含通用配置的配置时，必须由用户选择其一）。
2. 将阶段 2 合并后的节点集合写入该配置的 `proxies`（以及必要时的 `proxy-providers` 关联策略；具体策略见后续实现细化）。
3. 确保最终配置满足“完整配置”的定义（通用配置字段存在 + 节点集合可被后续修改引用）。

> 关键点：本路径不依赖模板生成通用配置；模板在此路径下不应被强制要求。

### 路径 B：使用模板 + subconverter 生成通用配置（当检测到“仅节点”，或用户主动选择）

适用：

- 中转信息检测为**仅节点**（此时模板必选：默认或自定义），或
- 中转信息虽包含通用配置，但用户选择**不使用**自己的通用配置，而改用模板生成。

产物：将“中转节点 + 用户节点（含落地） + 模板 + 参数”导入 subconverter，生成完整配置，作为后续修改基础。

处理步骤（要求语义）：

1. 选择模板：
   - 若 `transit.general_config? = false`：模板**必选**（可使用默认模板或用户自定义模板）。
   - 若 `transit.general_config? = true` 且用户仍选择路径 B：模板可选，但一旦选择路径 B，必须最终确定一个模板（默认或自定义）。
2. 组装 subconverter 输入：
   - 节点：来自阶段 2 合并后的节点集合（可能来自多份输入）
   - 模板：默认模板或用户自定义模板
   - 参数：subconverter 所需的最小参数集合（后续在 04 的 API 契约中固化）
3. 调用 subconverter 生成配置并解析产物，得到包含通用配置与节点落位的完整配置。

---

## 输出

- **完整配置（YAML 结构）**：作为 [03-modify-config](03-modify-config.md) 的输入。

### 失败与报错（最低要求）

- 当 `transit.general_config? = false` 且用户未提供模板（也未允许默认模板）时：必须报错（无法生成完整配置）。
- 当生成出的配置缺少通用配置最小字段集合（例如缺少 `external-controller`）时：必须视为生成失败并报错（不可进入后续修改阶段）。

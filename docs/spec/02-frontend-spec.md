# 02 - 前端 UI 规格

> 本章定义三阶段前端 UI 的结构、交互与业务逻辑。后端 API 契约见 03（待补）。

### Nav Bar

| 元素 | 说明 |
|------|------|
| Title | "Chain Converter for Mihomo" |

## 阶段 1：转换区

转换区是用户的入口，负责收集输入并调用 subconverter 生成完整配置。

### 1.1 落地节点信息输入区（左侧）

- **位置**：左侧
- **用途**：输入落地节点/订阅信息
- **输入方式**：
  - 直接粘贴订阅链接（URL）
  - 直接粘贴节点链接（`ss://`、`vmess://`、`trojan://`、`socks5://` 等）
  - 通过右上角**手动添加按钮**新增 SOCKS5 节点（见 1.1.1）
- **显示规则**：
  - 每条信息占一行
  - 左侧标记行号
  - 内容不换行，必要时输入窗口允许横向滚动
- **允许混合输入**：用户可在同一输入框内混合粘贴订阅 URL 与节点链接

#### 1.1.1 手动添加 SOCKS5 节点

- **触发**：落地信息区右上角的"添加 SOCKS5 节点"按钮
- **形态**：点击后弹出**卡片式表单**，标题为"添加 SOCKS5 节点"
- **表单字段**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| name | text | 是 | 节点名称 |
| server | text | 是 | 服务器地址 |
| port | number | 是 | 端口号 |
| username | text | 否 | 用户名 |
| password | text | 否 | 密码 |

- **提交行为**：
  - 用户点击确认后，将表单内容转换为一条新的**落地节点信息**
  - 新条目追加到落地信息输入区，表现为输入区中的新增一行
  - 前端可内部生成标准 `socks5://` 节点链接作为该行的实际值
- **协议限制**：当前仅支持手动添加 SOCKS5，不支持其他协议的手动表单

### 1.2 中转信息输入区（右侧）

- **位置**：右侧
- **用途**：输入中转节点/订阅信息，并按需开启端口转发服务输入
- **输入方式**：
  - 直接粘贴订阅链接（URL）
  - 直接粘贴节点链接
- **显示规则**：
  - 每条信息占一行
  - 左侧标记行号
  - 必要时输入窗口允许横向滚动
- **区域头部交互**：
  - 右上角提供**端口转发功能开关**（switch/slidebar）
  - 开关默认**关闭**
  - 开启后，在中转信息区下方展开独立的端口转发输入区域（见 1.2.1）
  - 关闭时，端口转发输入区域默认隐藏，且不参与本次提交

#### 1.2.1 端口转发服务信息

- **启用方式**：仅当右上角端口转发开关开启后显示
- **布局**：从中转信息区下方拆分出独立区域，不与中转订阅/节点混合在同一输入框
- **格式**：`server:port`（纯文本，如 `relay.example.com:1080`）
- **无需额外字段**（无名称、无协议等）
- **输入规则**：
  - 支持多行输入，每行一条端口转发服务信息
  - 与中转订阅/节点输入区分离维护
- **识别规则**：仅对该独立区域中的内容按 `server:port` 规则解析，无需再与节点链接/订阅 URL 混合判别

### 1.3 高级选项（右下角）

- **形态**：折叠区域（平时折叠隐藏），点击"高级选项"按钮展开
- **包含**：

| 选项 | 类型 | 说明 |
|------|------|------|
| 客户端 | 下拉菜单 | 当前仅 Clash/Mihomo |
| 转换模板 | 下拉菜单 | 默认模板 + 可扩展自定义模板 |
| Emoji | checkbox | 是否在节点名前加 emoji |
| 启用 UDP | checkbox | 是否启用 UDP |
| 跳过证书验证 | checkbox | 是否跳过 SSL 证书验证 |

### 1.4 自动识别按钮

- **样式**：大号横向按钮，视觉突出
- **功能**：
  1. 将左右两侧输入内容+高级选项参数提交后端
  2. 后端调用 subconverter 生成完整配置
  3. 前端解析返回结果：
     - **落地节点列表**：左侧输入的所有节点全部作为落地节点
     - **区域策略组**：从 subconverter 生成的 `proxy-groups` 中提取 6 个区域策略组（香港、美国、日本、新加坡、台湾、韩国），**剔除策略组中的所有落地节点**
     - **端口转发服务列表**：当端口转发开关开启时，从右侧独立端口转发输入区解析 `server:port` 条目
  4. 自动为每个落地节点匹配区域策略组，填充到配置区
  5. 引导用户平滑过渡到**阶段 2：配置区**

### 1.5 业务约束

- **禁止同源**：落地输入区与中转输入区**不允许使用同一份订阅**。前端可做 URL 重复检测并提示用户。

## 阶段 2：配置区

配置区以列表形式展示和编辑每条落地节点的链式代理/端口转发配置。

### 2.1 整体布局

- 类似表格/列表，每一 row 为一条配置
- 列结构：

| 列 | 内容 | 交互 |
|----|------|------|
| 第一列：落地节点 | 从转换区获取的落地节点逐行排列 | 可手动添加；提供节点列表选择（从 proxies 中拉取） |
| 第二列：中转方式 | `链式代理` 或 `端口转发`（互斥） | 可点击切换 |
| 第三列：前置/中转节点 | 自动匹配区域策略组或端口转发服务 | 自动填写；失败时手动选择 |

### 2.2 落地节点列（第一列）

- **数据来源**：阶段 1 自动识别后，左侧输入的所有节点全部作为落地节点
- **可手动添加**新的落地节点
- **提供节点列表选择**（从完整配置中的 `proxies` 拉取）

### 2.3 中转方式列（第二列）

- **默认值**：链式代理
- 可点击在 `链式代理` ↔ `端口转发` 之间切换
- **互斥约束**：一个落地节点不能同时应用链式代理和端口转发
- 切换后第三列的可选项应对应变化（策略组/节点 vs 端口转发服务）

### 2.4 前置/中转节点列（第三列）

#### 选择"链式代理"时

- **自动匹配逻辑**（详见 2.4.1 区域匹配规则）
- **可选来源**：
  - 策略组（proxy-groups 中的条目，已剔除落地节点）
  - 单个节点（proxies 中的条目）
- **协议限制**：如果落地节点是 reality 协议，**不支持链式代理**，给出用户警告提示

#### 选择"端口转发"时

- **可选来源**：阶段 1 右侧输入中识别的端口转发服务列表
- **自动填写规则**：
  - 仅有 1 个端口转发服务：默认自动填入
  - 有多个端口转发服务：由用户手动选择

#### 2.4.1 区域自动匹配规则

**区域分类**（6 类）：

| 区域ID | 区域名 | 默认模板策略组名 | 节点名匹配关键字（参考） |
|--------|--------|------------------|--------------------------|
| HK | 香港 | 🇭🇰 香港节点 | HK、HongKong、Hong Kong、香港、🇭🇰、港、深港、九龙 |
| US | 美国 | 🇺🇸 美国节点 | US、USA、United States、美国、🇺🇸、美、洛杉矶、硅谷 |
| JP | 日本 | 🇯🇵 日本节点 | JP、Japan、日本、🇯🇵、东京、大阪 |
| SG | 新加坡 | 🇸🇬 新加坡节点 | SG、Singapore、新加坡、🇸🇬、狮城 |
| TW | 台湾 | 🇼🇸 台湾节点 | TW、Taiwan、台湾、🇹🇼、🇼🇸、台 |
| KR | 韩国 | 🇰🇷 韩国节点 | KR、Korea、韩国、🇰🇷、首尔 |

> 完整关键字列表和正则参考 [Custom_Clash.ini](https://raw.githubusercontent.com/Aethersailor/Custom_OpenClash_Rules/refs/heads/main/cfg/Custom_Clash.ini) 中的 `custom_proxy_group` 定义。

**匹配算法**：

1. 从落地节点名称中提取区域关键字
2. 用关键字在 `proxy-groups` 中查找对应区域的策略组
3. 匹配结果处理：
   - **唯一匹配**：自动填入该策略组
   - **多个匹配**：**不自动填写**，给出警告日志，由用户手动选择
   - **无匹配**：不自动填写，由用户手动选择
4. **策略组净化**：匹配到的策略组需已**剔除所有落地节点**（即策略组成员仅保留中转节点）
5. **要求策略组非空**：如果剔除落地节点后策略组变为空，给出警告

**关键字匹配规则**（沿用旧代码逻辑）：

- **大小写不敏感**
- **英文关键字**使用**词边界匹配**（避免 "JP" 误匹配 "JPN" 以外的包含 JP 的字符串）
- **中文/emoji 关键字**使用**子串匹配**

### 2.5 每行高级选项

- **触发**：展开按钮，向左滑动展开高级选项列
- **包含**：

| 选项 | 说明 | 默认 |
|------|------|------|
| 替换 / 创建新节点 | 替换 = 直接修改原节点；创建 = 保留原节点，复制出新节点再修改 | 替换（直接修改） |
| 是否改名 | 允许用户自定义修改后的节点名 | 不改名 |
| 复制本行 | 复制当前行的配置到新行 | — |
| +/- 行 | 增加或删除配置行 | — |

### 2.6 生成按钮

- **样式**：大号横向按钮，视觉突出
- **功能**：将配置区的所有配置信息提交后端，应用链式代理/端口转发修改，生成最终 YAML
- 引导用户进入**阶段 3：生成区**

## 阶段 3：生成区

### 3.1 链接展示区（第一行）

- **长链接**：将所有参数编码到 URL query 中的生成链接（可直接粘贴到 Mihomo 客户端作为订阅地址）
- 可点击区域复制链接
- **右侧选项**：切换使用短链接（toggle/checkbox）
  - **短链接**：后端生成的 `/subscription/<id>.yaml` 形式

### 3.2 操作按钮（第二行）

三个大号按钮依次排列：

| 按钮 | 功能 |
|------|------|
| 打开 | 在新标签页打开 YAML 内容 |
| 复制 | 复制 YAML 内容到剪贴板 |
| 下载 | 下载为 `.yaml` 文件 |

### 3.3 日志区（最下方）

- 可折叠/展开
- 展示处理过程的日志信息（解析、生成、修改步骤的输出与警告）
- 用于排查问题和向用户展示处理细节

---

## 跨阶段交互与导航

- 三个阶段**依次展开**（或使用 stepper/tab 导航）
- 用户可以回退到前一阶段修改输入并重新执行
- 阶段间的数据传递通过前端状态管理（内存中）

---

## 错误与警告提示

### 必须报错（阻断）

- 落地输入或中转输入为空
- 落地与中转使用了同一份订阅 URL
- subconverter 生成失败
- 配置区中有未填写前置/中转节点的行

### 必须警告（不阻断，提示用户）

- 落地节点为 reality 协议且选择了链式代理
- 区域匹配到多个策略组，无法自动选择
- 策略组剔除落地节点后变为空
- 区域无法自动识别的落地节点

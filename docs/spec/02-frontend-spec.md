# 02 - 前端 UI 规格

> 本章定义三阶段前端 UI、前端状态与交互语义。后端接口契约见 [03-backend-api](03-backend-api.md)。

---

## 全局布局

### Nav Bar

| 元素 | 说明 |
|------|------|
| Title | `Chain Converter for Mihomo` |
| 亮/暗色主题切换 | 切换 light / dark mode |

- 阶段 1 到阶段 3 均采用纵向单列堆叠布局

- 移动端保持同样的纵向顺序，自上而下滚动操作

## 阶段 1：输入区

输入区负责收集原始输入，并调用后端完成“转换并自动识别”。

### 1.1 落地节点信息输入区

- **位置**：输入区最上方
- **用途**：输入落地节点或落地订阅信息
- **输入方式**：
  - 直接粘贴订阅链接（URL）
  - 直接粘贴节点链接（`ss://`、`vmess://`、`trojan://`、`socks5://` 等）
  - 通过右上角按钮手动添加 SOCKS5 节点
- **显示规则**：
  - 每条信息占一行
  - 左侧标记行号
  - 内容不自动换行，必要时输入窗口允许横向滚动
  - 允许在同一输入框内混合粘贴订阅 URL 与节点链接

#### 1.1.1 手动添加 SOCKS5 节点

- **触发**：落地信息区右上角的“添加 SOCKS5 节点”按钮
- **形态**：点击后弹出卡片式表单，标题为“添加 SOCKS5 节点”

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `name` | text | 是 | 节点名称 |
| `server` | text | 是 | 服务器地址 |
| `port` | number | 是 | 端口号 |
| `username` | text | 否 | 用户名 |
| `password` | text | 否 | 密码 |

- **提交行为**：
  - 用户点击确认后，将表单内容转换为一条新的落地节点信息
  - 新条目追加到落地信息输入区，表现为输入区中的新增一行
  - 新条目以标准 `socks5://` URI 单行文本写回输入区
- **校验规则**：
  - `server` 仅支持 IPv4 地址或域名，不支持 IPv6
  - `port` 不额外限制取值范围，由后端/底层能力决定是否可用
  - `username` 与 `password` 必须成对出现；只填写其中一个时阻止提交
- **协议限制**：当前仅支持手动添加 SOCKS5，不支持其他协议的手动表单

#### 1.1.2 落地副本约定

- 阶段 2 不提供“复制本行”“新增行”“创建新节点”“重新选择落地节点”的能力
- 如需同一落地节点的多个副本，必须在阶段 1 的落地输入区自行复制多条 URI，并按需修改名称
- 若落地输入区识别到多条**完全一致**的 URI，系统应将其视为用户显式创建的副本，而不是去重删除
- 这类副本在识别后的节点名称上自动追加 ` 02`、` 03` 等后缀，以保证阶段 2 中的 `landingNodeName` 唯一且稳定

### 1.2 中转信息输入区

- **位置**：落地信息输入区下方
- **用途**：输入中转节点或中转订阅信息
- **输入方式**：
  - 直接粘贴订阅链接（URL）
  - 直接粘贴节点链接
- **显示规则**：
  - 每条信息占一行
  - 左侧标记行号
  - 内容不自动换行，必要时输入窗口允许横向滚动

#### 1.2.1 端口转发服务信息

- **启用前提**：
  - 用户先展开默认折叠的高级菜单
  - 再开启其中的“启用端口转发（实验性）”开关
- **默认状态**：默认隐藏，不占据主界面空间
- **布局**：显示时位于中转信息区下方、高级菜单区上方；为独立区域，不与中转订阅或节点混合在同一输入框
- **格式**：`server:port`，如 `relay.example.com:1080`
- **字段约束**：仅接受 `server:port`，不支持备注名或其他附加字段
- **输入规则**：
  - 支持多行输入，每行一条端口转发服务信息
  - 与中转订阅或节点输入区分离维护
  - 多个条目保留输入顺序，供阶段 2 展示和选择
- **校验规则**：
  - 仅支持 IPv4 地址或域名，不支持 IPv6
  - `port` 不额外限制取值范围，由后端/底层能力决定是否可用
  - 重复条目执行去重并警告

### 1.3 高级菜单区

- **位置**：位于中转信息区下方；当端口转发服务信息区显示时，位于其下方
- **形态**：默认折叠隐藏，点击“高级菜单”按钮展开

| 选项 | 类型 | 说明 |
|------|------|------|
| 客户端 | 下拉菜单 | 当前仅 Clash/Mihomo |
| 转换模板 | 只读展示或单选项下拉 | 当前仅支持默认模板；自定义模板能力只预留接口 |
| Emoji | checkbox | 是否在节点名前加 emoji |
| 启用 UDP | checkbox | 是否启用 UDP |
| 跳过证书验证 | checkbox | 是否跳过 SSL 证书验证 |
| 启用端口转发（实验性） | switch | 默认关闭；开启后显示独立的端口转发服务输入区 |

### 1.4 转换按钮

- **样式**：大号横向按钮，视觉突出
- **位置**：输入区最下方
- **按钮文案**：`转换并自动识别`
- **功能**：
  1. 将落地输入区、中转输入区和高级菜单参数提交后端；端口转发服务信息不传入 subconverter
  2. 后端调用 subconverter 生成完整配置
  3. 后端返回阶段 2 所需的最小产物：
     - `completeConfig`：subconverter 转换后的完整配置
     - `landingNodes[]`：按落地输入来源识别出的落地节点列表
     - `forwardRelays[]`：端口转发服务列表
  4. 前端基于 `landingNodes[]` 为阶段 2 生成默认行，并尝试自动匹配链式代理目标
  5. 引导用户进入阶段 2

### 1.5 阶段 1 业务约束

- 布局顺序固定为：落地信息 -> 中转信息 -> 条件显示的端口转发服务信息 -> 默认折叠的高级菜单 -> 转换按钮
- 仅展开高级菜单本身不会显示端口转发输入区；必须继续开启端口转发开关后才显示
- 不再对落地输入区与中转输入区做订阅同源判定、URL 归一化比较或短链跳转归并判断

## 阶段 2：配置区

配置区按“每个落地节点一行”的固定模型展示链式代理或端口转发配置。

### 2.1 行模型

阶段 2 行数据结构固定为：

| 字段 | 类型 | 说明 |
|------|------|------|
| `rowId` | string | 前端内部稳定标识，用于渲染、定位和错误提示 |
| `landingNodeName` | string | 本行绑定的落地节点名称 |
| `mode` | enum | `none` / `chain` / `port_forward` |
| `targetName` | string \| null | 第三列选择值；其语义由 `mode` 决定 |

补充约束：

- 阶段 1 自动识别后，默认每个落地节点生成一条配置行
- 同一个落地节点在阶段 2 **不能出现多行**
- 阶段 2 不提供以下操作：
  - 重新选择落地节点
  - 复制本行
  - `+/-` 行
  - 创建新节点
- 因而阶段 2 不存在“后写覆盖前写”问题；每个落地节点最多只有一份配置结果

### 2.2 整体布局

- 配置区表现为类似表格的列表，最左侧标记行号或序号

| 列 | 内容 | 交互 |
|----|------|------|
| 第一列：落地节点 | `landingNodeName` | 只读展示，不可重新选择 |
| 第二列：配置方式 | `mode` | 在 `无需配置`、`链式代理`、`端口转发` 三者之间切换 |
| 第三列：目标 | `targetName` | 根据 `mode` 展示对应选择器 |

### 2.3 第一列：落地节点

- **数据来源**：阶段 1 返回的 `landingNodes[]`
- **展示语义**：只展示本行绑定的落地节点名称
- **禁止行为**：
  - 不允许手动输入节点链接或节点参数
  - 不允许切换为其他落地节点
  - 不允许在阶段 2 新增自由节点

### 2.4 第二列：配置方式

- **可选值**：
  - `无需配置`
  - `链式代理`
  - `端口转发`
- **默认值**：
  - 若自动匹配到唯一链式代理目标，则默认 `mode = chain`
  - 否则默认 `mode = none`
- **切换规则**：
  - `mode = none` 时，第三列清空并禁用
  - `mode = chain` 时，第三列切换为“策略组或单个节点”选择器
  - `mode = port_forward` 时，第三列切换为“端口转发服务”选择器
- **显示门槛**：
  - 未开启端口转发（实验性）时，不显示 `端口转发` 选项
  - 已开启但未识别到任何端口转发服务时，同样不显示 `端口转发` 选项

### 2.5 第三列：目标

`targetName` 的语义必须稳定，不允许“一套 UI、两套后端含义”。

#### 当 `mode = none`

- `targetName = null`
- UI 显示为空或禁用态
- 生成时该落地节点保持原样，不做任何链式代理或端口转发改写

#### 当 `mode = chain`

- **可选来源**：
  - 策略组：`proxy-groups` 中的条目，且已剔除落地节点
  - 单个节点：`proxies` 中的条目
- **自动匹配逻辑**：优先按 2.5.1 的区域规则尝试自动匹配
- **最终语义**：无论用户选中的是策略组还是单个节点，生成时都直接写入落地节点的 `dialer-proxy`
- **协议限制**：若落地节点被判定为 `vless-reality`，则不支持链式代理；应提示用户改为 `无需配置` 或 `端口转发`
- **建议性提示**：若落地节点端口号大于等于 `10000`，仅给出推荐提示，说明链式代理更推荐使用 `10000` 以内端口；该提示不阻断生成，也不计入 warning

#### 当 `mode = port_forward`

- **可选来源**：阶段 1 返回的 `forwardRelays[]`
- **自动填写规则**：
  - 仅有 1 个端口转发服务时，切换到 `端口转发` 后自动填入
  - 有多个端口转发服务时，由用户手动选择
  - 展示顺序保持阶段 1 输入顺序
- **最终语义**：生成时以选中的 `targetName` 定位端口转发服务，并直接在原落地节点上替换 `server` 与 `port`
- **协议说明**：`vless-reality` 在端口转发下允许直接修改原节点，不再要求复制新节点
- **字段修改范围**：仅替换 `server` 与 `port`，不联动修改 `sni`、`host`、`alpn` 等其他字段

#### 2.5.1 区域自动匹配规则

| 区域 ID | 区域名 | 默认模板策略组名 | 节点名匹配关键字（参考） |
|---------|--------|------------------|--------------------------|
| `HK` | 香港 | `🇭🇰 香港节点` | HK、HongKong、Hong Kong、香港、🇭🇰、港、深港、九龙 |
| `US` | 美国 | `🇺🇸 美国节点` | US、USA、United States、美国、🇺🇸、美、洛杉矶、硅谷 |
| `JP` | 日本 | `🇯🇵 日本节点` | JP、Japan、日本、🇯🇵、东京、大阪 |
| `SG` | 新加坡 | `🇸🇬 新加坡节点` | SG、Singapore、新加坡、🇸🇬、狮城 |
| `TW` | 台湾 | `🇼🇸 台湾节点` | TW、Taiwan、台湾、🇹🇼、🇼🇸、台 |
| `KR` | 韩国 | `🇰🇷 韩国节点` | KR、Korea、韩国、🇰🇷、首尔 |

> 完整关键字列表和正则参考 [Custom_Clash.ini](https://raw.githubusercontent.com/Aethersailor/Custom_OpenClash_Rules/refs/heads/main/cfg/Custom_Clash.ini) 中的 `custom_proxy_group` 定义。

匹配算法：

1. 从落地节点名称中提取区域关键字
2. 用关键字在 `proxy-groups` 中查找对应区域的策略组
3. 匹配结果处理：
   - 唯一匹配：自动填入该策略组，并设置 `mode = chain`
   - 多个匹配：不自动填写，给出 warning，由用户手动选择
   - 无匹配：不自动填写，保留 `mode = none`
4. 匹配到的策略组需已剔除所有落地节点
5. 若剔除后策略组为空，给出 warning

关键字匹配规则：

- 大小写不敏感
- 英文关键字使用词边界匹配
- 中文和 emoji 关键字使用子串匹配
- 同一落地节点名称若同时命中多个区域的英文关键字，视为区域识别失败，不做自动填写

### 2.6 生成按钮

- **样式**：大号横向按钮，视觉突出
- **功能**：将阶段 1 输入快照与阶段 2 配置快照一起提交后端，生成最终 YAML
- **成功后**：进入阶段 3

## 阶段 3：输出区

### 3.1 链接展示区

- **长链接**：编码“阶段 1 输入 + 阶段 2 配置快照”的订阅链接
- **短链接**：后端生成的长链接别名，形如 `/subscription/<id>.yaml`
- **切换展示**：前端可提供“使用短链接”开关，但长链接始终是规范化源数据
- **反向恢复**：
  - 输入长链接时，前端或后端应能反解出阶段 1 输入与阶段 2 配置快照
  - 输入短链接时，后端先解析出其绑定的长链接，再返回同一份快照用于页面回填

### 3.2 操作按钮

| 按钮 | 功能 |
|------|------|
| 打开 | 在新标签页打开 YAML 内容 |
| 复制 | 复制 YAML 内容到剪贴板 |
| 下载 | 下载为 `.yaml` 文件 |

### 3.3 消息区

- 可折叠或展开
- 展示后端返回的结构化消息
- 前端统一按 `level` 渲染样式
- 推荐消息结构：`level`、`code`、`message`、`context`

---

## 跨阶段交互与导航

- 三个阶段依次展开，或使用 stepper/tab 导航
- 用户若修改阶段 1 输入，必须重新执行“转换并自动识别”
- 重新转换后，阶段 2 按新的 `landingNodes[]` 重新生成默认行，不保留旧的手工选择结果
- 阶段间的数据传递通过前端状态管理完成；阶段 3 的长链接可作为外部可复现快照

---

## 错误与提示

### 阻断错误

- 落地输入为空或中转输入为空
- subconverter 生成失败
- 某一行选择了 `链式代理` 或 `端口转发`，但第三列未填写

### Warning

- 落地节点被判定为 `vless-reality` 且选择了 `链式代理`
- 区域匹配到多个策略组，无法自动选择
- 策略组剔除落地节点后变为空

### 建议提示

- 落地节点选择链式代理时，其端口号大于等于 `10000`

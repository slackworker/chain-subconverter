# 00 - 治理与总则

## 文档定位

本文档定义项目治理规则、spec 权威顺序与 spec 编写原则。

## 当前阶段

本项目处于 **spec-driven 的彻底重构阶段**。

## 核心规则

- 任何既有架构、代码、逻辑、文档与历史决策都可以被质疑、替换或重写。
- 既有实现仅作参考，不作为正确性依据。
- spec 出现歧义、遗漏或隐含假设时，必须先澄清，再设计或实现。
- 鼓励提出更优实现、最佳实践与必要的结构调整。
- 一旦 spec 结论确认，以其作为唯一准绳。
- spec 文档职责必须清晰，同一问题只在一个位置定义，其他文档仅引用。
- spec 内容力求简洁，只写怎么做；非必要不展开背景、论证、示例与不怎么做。
- 上位文档已定义的约束，下位文档不重复，只有在需要细化时才补充新增信息。

## 权威顺序

1. 已确认的 `docs/spec/*.md`
2. 本文档
3. 现有代码、旧文档与历史决策

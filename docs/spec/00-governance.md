# 00 - 治理与总则

## 当前阶段

本项目处于 **spec-driven 的彻底重构阶段**。既有实现仅作参考。

## 权威顺序

冲突时按以下优先级裁决：

1. `docs/spec/01–05` 各 spec 文档
2. 本文档（治理与编写原则）
3. 现有代码、旧文档与历史决策

## 核心规则

- spec 出现歧义、遗漏或隐含假设时，必须先澄清，再设计或实现。
- 任何既有架构、代码或历史决策都可以被质疑、替换或重写。
- 鼓励提出更优实现、最佳实践与必要的结构调整。

## 编写原则

- **职责单一**：同一问题只在一个文档定义，其他文档仅引用。
- **层级递进**：上位文档已定义的约束，下位文档只补充细化信息。
- **简洁直接**：只写结论和做法，不展开背景、论证或示例。

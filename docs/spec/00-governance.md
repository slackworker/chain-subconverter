# 00 - 治理与总则

## 当前阶段

本项目处于 **收口与微调阶段**，以实现与文档对齐整理为主，按需做小范围修订并保持二者同步。

- 已确认文档与当前实现共同作为收口与评审依据。
- 对齐整理过程中，优先消除文档与实现的不一致。

## 权威顺序

冲突时按以下优先级裁决：

1. 本文档（治理与编写原则）
2. `docs/spec/01–05` 各 spec 文档
3. 现有代码、旧文档与历史决策

- `00-governance` 是最高原则。
- `01–05` 是各自主题范围内的权威定义。

## 核心规则

- 文档出现歧义、遗漏或隐含假设时，必须先澄清，再做实现或文档修订。
- 任何既有架构、代码或历史决策都可以被质疑、替换或重写。
- 鼓励提出更优实现、最佳实践与必要的结构调整。
- 经澄清确认且需要固化的结论，必须同步更新对应文档与实现。

## 同步机制

- 发现实现与文档冲突时，先请求澄清；确认后的必要结论同步修订文档与实现。

## 编写原则

- **职责单一**：同一问题只在一个文档定义，其他文档仅引用。
- **层级递进**：上位文档已定义的约束，下位文档只补充细化信息。
- **简洁直接**：只写结论和做法，不展开背景、论证或示例。

## 文档入口

- 文档导航、阅读顺序与索引统一维护在 `docs/README.md`。

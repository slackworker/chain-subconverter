# 00 - 治理与总则

## 当前阶段

本项目处于 **spec 与开发并行推进阶段**，以已确认 spec 推进实现，并保持二者同步。

- 已确认的 spec 仍是当前实现与评审的直接依据。
- 既有实现与旧文档仅作参考，不作为裁决依据。

## 权威顺序

冲突时按以下优先级裁决：

1. `docs/spec/01–05` 各 spec 文档
2. 本文档（治理与编写原则）
3. 现有代码、旧文档与历史决策

## 核心规则

- spec 出现歧义、遗漏或隐含假设时，必须先澄清，再设计或实现。
- 任何既有架构、代码或历史决策都可以被质疑、替换或重写。
- 鼓励提出更优实现、最佳实践与必要的结构调整。
- 经澄清确认且需要固化的实现结论，必须更新到对应 spec。

## 同步机制

- **先澄清，后落地**：涉及规则变化时先确认边界。
- **双向同步**：spec 驱动实现；必要结论回写 spec。
- **最小回写**：只记录必要约束、行为变化与决策结论。
- **同轮收口**：同一变更尽量同时更新实现与相关 spec。

## 编写原则

- **职责单一**：同一问题只在一个文档定义，其他文档仅引用。
- **层级递进**：上位文档已定义的约束，下位文档只补充细化信息。
- **简洁直接**：只写结论和做法，不展开背景、论证或示例。

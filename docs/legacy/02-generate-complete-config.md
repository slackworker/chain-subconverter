# 02 - 生成完整配置（统一走模板 + subconverter）

> 本章定义“如何得到可用于后续改写的**完整配置**”。后续链式代理/端口转发只在“完整配置”上进行（见 [03-modify-config](03-modify-config.md)）。

## 术语（本章口径）

- **节点集合（Nodes）**：用户提供或解析得到的中转节点与落地节点，最终写入 `proxies`（以及可选 `proxy-providers`）。
- **通用配置（GeneralConfig）**：使 Mihomo 可运行并支撑后续改写的最小非节点结构，至少应包含 `mixed-port` 或 `port`、`mode`、`external-controller`、`proxy-groups`、`rules`。
- **模板（Template）**：用于让 subconverter 生成通用配置的来源。当前主路径使用默认模板；自定义模板作为后续扩展能力。
- **完整配置（CompleteConfig）**：`节点集合 + 通用配置`，并满足：
  - 顶层包含通用配置最小字段集合
  - 顶层存在 `proxies`（或 `proxy-providers`）
  - 同时覆盖后续修改阶段需要引用的中转节点与落地节点
- **角色元信息（Role Metadata）**：完整配置之外仍需保留的业务信息，包括：
  - `selected_transit_names[]`
  - `selected_landing_names[]`
  - `landing_included_in_transit`
  - `forward_relays[]`
  - 以及后续修改所需的用户选择结果
- **subconverter**：用于“节点 + 模板 + 少量参数 → 生成完整配置”。

---

## 设计原则

本阶段采用**单一路径**：

- 不再提前判断“中转输入是否包含通用配置”。
- 不再提供“直接复用用户原始通用配置”的分流路径。
- 无论用户输入的是订阅、YAML、节点链接还是表单节点，只要能够抽取到所需节点，统一进入 **模板 + subconverter** 生成流程。

前端的职责不是替用户做复杂判定，而是**把 subconverter 真正需要的输入收集完整**。

---

## 输入

输入在概念上分为五类（具体 API 结构见 [04-output-and-api](04-output-and-api.md)）：

- **中转信息输入（必填）**：订阅、YAML、节点链接或表单节点；用于提供中转节点候选。
- **落地信息输入（条件必填）**：订阅、YAML、节点链接或表单节点；用于提供落地节点候选。
- **落地已包含在中转中（必填布尔）**：若为 `true`，表示用户确认落地节点可直接从中转输入中选出，此时可不再单独提供 `landing_source`。
- **模板（可选）**：默认使用内置模板；后续可扩展用户自定义模板。
- **端口转发服务信息（可选）**：仅当后续要做端口转发时需要，本章负责接收并保留给下一阶段。

> 产品语义上，系统仍要求形成“中转”和“落地”两类信息。通常表现为至少两份节点/订阅信息；若用户勾选“落地已包含在中转中”，则允许只提供一份来源，再从中显式划分两类角色。

---

## 阶段 1：解析与前端引导（输入 → 可选节点）

本阶段允许解析输入、提取节点、给前端做展示和勾选，但**不再承担“通用配置判定”职责**。

要求输出语义一致的结构化结果：

- `transit.nodes[]`：从中转输入解析到的节点集合
- `landing.nodes[]`：从落地输入解析到的节点集合
- `template.selected`：默认模板或用户自定义模板；当前若未指定则默认使用内置模板
- `subconverter.params`：少量可选参数
- `forward_relays[]`：用户录入的端口转发服务信息

补充要求：

- 对 URL 类型输入，若返回内容可判定为 Base64 编码文本，必须先解码，再继续解析。
- 若 URL 返回内容不是 Base64，则按原始内容继续解析。
- 即使输入本身是完整 Mihomo YAML，也只将其视为一种可解析节点来源；本阶段不再据此走特殊生成分支。

---

## 阶段 2：角色确认、节点合并与命名冲突处理

系统需要把多来源节点整理为一份可交给 subconverter 的统一节点集合，同时保留业务角色信息。

### 2.1 角色确认

以下信息必须由用户显式提供或确认：

- 哪些节点属于中转：`selected_transit_names[]`
- 哪些节点属于落地：`selected_landing_names[]`
- 落地是否已包含在中转输入中：`landing_included_in_transit`

规则：

- 自动识别只能作为前端默认建议，最终角色划分必须以用户确认结果为准。
- 若 `landing_included_in_transit = true`，则落地节点可来自 `transit.nodes[]`。
- 若 `landing_included_in_transit = false`，则必须提供独立的 `landing_source` 或等价落地节点输入。

### 2.2 节点合并

用于生成完整配置的节点集合应来自：

- 中转输入解析出的节点
- 落地输入解析出的节点（若有）
- 必要时用户手动补充的节点

合并要求：

- 写入 `proxies` 的节点 `name` 必须唯一；若冲突，必须重命名，且不可静默覆盖。
- 对生成完整配置而言，最终节点集合必须至少覆盖全部已确认的中转节点与落地节点。
- 对下一阶段而言，角色元信息必须独立保存，不能只依赖节点命名约定反推。

---

## 阶段 3：组装 subconverter 输入

本阶段统一按 subconverter 视角构造输入，不再存在路径 A / 路径 B 分流。

最小语义要求：

1. 选择模板：
   - 当前默认使用内置模板
   - 后续若支持自定义模板，仍应落在同一流程中
2. 组装节点输入：
   - 输入节点应至少包含全部已确认的中转节点与落地节点
   - 节点来源可来自多份输入，但对 subconverter 来说应表现为统一节点集
3. 传递少量 subconverter 参数：
   - 参数范围应尽量收敛，避免前端暴露过多底层细节
   - 具体字段在 04 的 API 契约中固化
4. 调用 subconverter：
   - 生成带有通用配置和节点落位的完整配置

---

## 阶段 4：生成结果校验与保留元信息

生成出的完整配置至少需要满足：

- 顶层包含通用配置最小字段集合
- 顶层存在 `proxies` 或 `proxy-providers`
- `selected_transit_names[]` 与 `selected_landing_names[]` 中涉及的节点，在完整配置中可被后续阶段稳定引用

除完整配置本体外，还必须保留以下信息，供 [03-modify-config](03-modify-config.md) 使用：

- `selected_transit_names[]`
- `selected_landing_names[]`
- `landing_included_in_transit`
- `forward_relays[]`
- 用户后续对链式代理和端口转发的选择结果

---

## 输出

- **完整配置（YAML 结构）**：作为 [03-modify-config](03-modify-config.md) 的输入。
- **角色元信息**：与完整配置绑定保存，供下一阶段继续修改。

### 失败与报错（最低要求）

- 当无法形成有效的“中转 + 落地”角色集合时：必须报错。
- 当 `landing_included_in_transit = false` 且用户未提供任何落地信息时：必须报错。
- 当 subconverter 生成失败时：必须报错。
- 当生成出的配置缺少通用配置最小字段集合（例如缺少 `external-controller`）时：必须视为生成失败并报错。

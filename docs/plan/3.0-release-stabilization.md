# 3.0 发布整理 Plan

本文是当前阶段唯一执行计划，目标是把分支/标签、工作流、文档、发布回归和固定测试基线收口到可维护状态。实时进度统一维护在 [../progress/STATUS.md](../progress/STATUS.md)。

## 1. 当前基线

- 当前默认入口固定为 `/`，绑定 `default` scheme；`/ui/a`、`/ui/b`、`/ui/c` 保留为实验入口。
- 当前分支模型收口为 `dev / beta / main`：`dev` 用于开发与手动 `dev-latest`，`beta` 用于预发布与 `beta-latest`，`main` 负责当前公开滚动镜像 `latest`。
- 当前阶段先不强制引入版本号标签；Beta / 正式版再补 `vX.Y.Z-beta.N` 与 `vX.Y.Z` 规则。
- 当前唯一稳定 fixture 仍是 [../testing/3pass-ss2022-test-subscription](../testing/3pass-ss2022-test-subscription.md) 对应的 `internal/review/testdata/3pass-ss2022-test-subscription/`。

## 4. 推荐推进顺序

1. 先完成 W0 和 W2，彻底移除旧发布模型的当前地位。
2. 再完成 W3，把发布回归与反馈闭环固定下来。
3. 最后做 W4，把固定测试流程扩成后续自动化的共享基线。

## 5. 不在本轮范围

- 不做新的大功能开发；重点是发布整理与可维护性交付。
- 不在当前阶段强制设计完整版本号体系；Beta / 正式版再补齐。
- 不重写 spec 02–04 的业务规则，只同步当前实现与执行边界。
- 不在本轮强制把 `subconverter` 浮动 tag 替换为 sha256 digest；仅要求回归记录能说明当前使用来源。
